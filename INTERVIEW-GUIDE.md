## 🎯 面试问答指南

### Q: 这个项目解决了什么问题？

**回答思路：**
> 「做 AIGC 和 AI Agent 的团队，每周都要手动刷一遍 Dify、Coze、LangChain 等竞品的更新，耗时且容易遗漏。我把它做成一个自动化的 AI Agent：配置竞品关键词 → 自动爬取 → AI 总结 → 定时推送，把每周 1-2 小时的手工工作降到零。」

### Q: 为什么说是 AI Agent，不只是个爬虫？

**回答思路**（这是关键区分点）：

| 爬虫 | AI Agent |
|------|----------|
| 固定 URL + 正则提取 | **自主搜索** — 用关键词查找相关来源 |
| 原始文本 | **AI 分析** — 分类、提取关键信息、判断影响 |
| 数据 dump | **结构化简报** — 多竞品对比、趋势分析、行动要点 |
| 手动看 | **主动推送** — 定时触达、无需等待 |

> 「它有四个 Agent 特征：**感知**（搜索和爬取信息）、**规划**（按简报模板组织内容）、**行动**（使用多种工具）、**记忆**（写入日志文件做持久化）。它不是一个被动脚本，而是一个有自主行为的 AI 系统。」

### Q: 架构怎么设计的？

```
┌─────────────────────────────────────────────────────┐
│                   用户配置层                          │
│           config.yaml (竞品/关键词/推送)              │
└──────────────────┬──────────────────────────────────┘
                   │
┌──────────────────▼──────────────────────────────────┐
│             调度层 (Cron Job)                        │
│         每周一 09:00 Asia/Shanghai                    │
└──────────────────┬──────────────────────────────────┘
                   │
┌──────────────────▼──────────────────────────────────┐
│             执行层 (Agent)                           │
│  ┌─────────┐ ┌──────────┐ ┌──────────────────┐    │
│  │ web_search │ web_fetch │  LLM 分析总结    │    │
│  └─────────┘ └──────────┘ └──────────────────┘    │
└──────────────────┬──────────────────────────────────┘
                   │
┌──────────────────▼──────────────────────────────────┐
│             交付层                                    │
│  日志文件 → Heartbeat 检测 → 飞书私聊推送           │
└─────────────────────────────────────────────────────┘
```

> 「三层架构：**配置层**解耦竞品信息与代码逻辑；**执行层**由 LLM 驱动的 Agent 自主完成信息搜集、分析、格式化；**交付层**通过文件信号机制实现隔离执行环境到主会话的消息推送。」

### Q: 技术栈和关键决策？

| 技术 | 用途 |
|------|------|
| **OpenClaw Agent Runtime** | Agent 运行框架（cron 调度 + 工具系统 + 消息通道） |
| **Web Search / Fetch** | 实时竞品信息采集 |
| **LLM (DeepSeek v4 Flash)** | AI 分析与简报生成（低成本高速度） |
| **Feishu API** | 即时消息推送 |
| **Cron + Heartbeat** | 定时调度与可靠交付（双重保障） |
| **YAML Config** | 零代码配置竞品和推送策略 |

> **关键决策说明：**
> 1. **隔离执行**：用 cron 隔离会话运行 Agent，避免阻塞主会话
> 2. **文件信号机制**：(cron 写文件 → heartbeat 推送)，解决隔离环境无法直接调用消息工具的问题
> 3. **中英双语搜索**：每个竞品同时搜中文和英文关键词，覆盖国内外动态
> 4. **结果去重**：同一竞品的多关键词搜索结果自动去重

### Q: 遇到什么挑战？怎么解决的？

这是面试最加分的部分：

**挑战 1：隔离环境缺少消息工具**
> 「Agent 在 cron 隔离会话中运行时没有 `message` 工具权限，无法直接推飞书。我设计了 **文件信号机制**：Agent 写完简报后创建 `.pending_delivery` 标记文件，主会话在下次 heartbeat 时检测到标记，代发消息。做到了执行与交付的解耦。」

**挑战 2：Agent 工具可用性不一致**
> 「隔离环境缺少 `web_search` 工具，改为全部通过 `web_fetch` 从固定来源（GitHub Releases、技术新闻站点）获取。后来发现用 `web_fetch` 反而更可控 —— 可以针对每个竞品精确拉取 Release Notes。」

**挑战 3：简报质量把控**
> 「早期版本 LLM 输出的简报要么太长要么太泛。我在 `briefing-agent.md` 中定义了严格的指令：限制简报 2000 字以内、要求格式模板化、强制每段包含「影响分析」标签。最终输出结构稳定、可读性强。」

### Q: 怎么保证可靠性？

> 「设计了**三级保障**：
> 1. **Cron 定时触发** — 周一 09:00 精确启动
> 2. **日志持久化** — 简报写入文件，重启不丢失
> 3. **Heartbeat 兜底推送** — 即使 cron 执行完无人在线，下次 heartbeat 也会把简报推出去
> 
> 另外，单个竞品抓取失败**不影响**其他竞品，做了部分容错。」

## 📈 项目亮点

1. **真正的 AI Agent** — 不是固定脚本，而是有感知/规划/行动的 Autonomous Agent
2. **生产级调度** — cron 定时 + heartbeat 兜底，双重可靠
3. **产品思维** — 从需求分析到配置化的用户界面，不只是技术自嗨
4. **全链路闭环** — 配置 → 采集 → 分析 → 推送，完整的数据流
5. **可扩展架构** — 加竞品改 YAML 就行，加推送渠道改配置就行
